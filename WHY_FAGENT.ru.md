# Почему fagent

[English version](./WHY_FAGENT.md)

Этот документ объясняет, в чём `fagent` сильнее более широких агентных платформ и почему его архитектура оптимизирована под долгую continuity-работу, а не только под one-shot чат.

## Коротко

`fagent` подходит тем, кому нужен:

- рантайм, который можно читать и дебажить напрямую
- память, которая больше, чем история сообщений
- graph-aware recall для связей и блокеров
- меньше waste основной модели в tool-heavy сценариях
- локальный способ посмотреть, что именно лежит в memory graph

Он не пытается быть самой широкой consumer-платформой. Он пытается быть сфокусированным рантаймом, который сохраняет связность между сессиями.

## Где fagent выигрывает

### 1. Меньше операционная поверхность, выше прозрачность

`fagent` строится вокруг одного Python package, одного config tree, одной workspace-модели и одной CLI-поверхности. За счёт этого проще:

- понять поведение рантайма
- дебажить retrieval и tool execution
- менять provider wiring
- инспектировать память прямо на диске

### 2. Память здесь многослойная, а не монолитная

Многие агенты останавливаются на одном из вариантов:

- только chat history
- embeddings плюс chat history
- периодические summaries

`fagent` сочетает:

- file memory для прозрачности
- vector memory для semantic recall
- graph memory для сущностей и связей
- workflow state для continuity активного выполнения
- task graph state для целей, блокеров и решений
- experience memory для повторяющихся recoveries
- shadow context для сжатия перед main-model вызовом
- routing, который меняет retrieval по типу запроса

### 3. Graph recall решает другую задачу, чем vector recall

Векторный поиск хорош для semantic similarity. Он слабее на вопросах вроде:

- что от чего зависит
- какое решение связано с блокером
- какой workflow node соединён с фактом
- какие сущности должны оставаться в одном task state

`fagent` относится к таким вопросам как к графовой задаче, а не только к задаче text similarity.

### 4. Локальный Graph UI — практическое преимущество

Во многих агентных системах graph memory невидима. `fagent` открывает её через локальный Graph UI server/editor.

Это позволяет:

- смотреть graph nodes и edges в браузере
- проверять, дала ли extraction полезную структуру
- дебажить качество памяти, а не гадать
- понимать, почему произошёл relationship-oriented recall

### 5. Основная модель получает guidance, а не сырую свалку

`fagent` умеет достать память из нескольких хранилищ и сжать её в shadow brief до запуска основной модели.

Это меняет экономику reasoning:

- меньше токенов на поиск по памяти
- меньше мусора от нерелевантных recall-артефактов
- меньше стартовых шагов вида «сейчас разберусь, что происходит»
- больше прямого reasoning по уже отфильтрованным данным

### 6. Tool-heavy работа не заставляет сильнейшую модель чинить всё подряд

В `fagent` есть отдельный workflow tool и роль `workflowLight`.

За счёт этого рантайм может:

- исполнять ordered tool chains
- интерпретировать rough workflow steps
- ремонтировать invalid steps и tool mismatches
- эскалировать к основной модели только когда это действительно нужно

### 7. Subagents дают ограниченную делегацию

`fagent` также поддерживает subagents для фонового выполнения. Это даёт главному рантайму способ делегировать сфокусированные задачи, не превращая всё в один длинный поток рассуждений.

## Почему такая память лучше простых memory-стеков

Преимущество не в том, что компонентов просто больше. Преимущество в том, что каждый компонент покрывает свой failure mode.

- file memory даёт auditability и manual inspection
- vector memory ловит fuzzy semantic matches
- graph memory хранит отношения
- workflow memory сохраняет execution state
- task graph memory держит живой структуру целей
- experience memory перестаёт заново находить один и тот же фикс
- shadow context уменьшает waste основной модели
- routing снижает долю нерелевантного retrieval

## Как понимать это сравнение

Смысл не в том, что `fagent` универсально лучше больших платформ. Смысл в том, что он оптимизирован под другой тип нагрузки:

- повторяющаяся проектная работа
- tool-heavy engineering flows
- memory-sensitive поведение агента
- операторов, которым важно видеть и настраивать рантайм
