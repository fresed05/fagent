# Почему fagent

Этот документ объясняет, в чём `fagent` сильнее более широких агентных стеков вроде [OpenClaw](https://github.com/openclaw/openclaw), как устроена память и почему встроенный инструмент последовательных workflow даёт практическое преимущество.

## Коротко

`fagent` выигрывает там, где нужен агент, который:

- легко читать и дебажить
- просто настраивать
- хорошо держит continuity между реальными рабочими сессиями

Это другая оптимизация по сравнению с OpenClaw. OpenClaw позиционируется как более широкий экосистемный продукт: desktop, mobile, web, расширения, более крупная distribution surface. Это мощно, но влечёт больше слоёв, больше moving parts и более тяжёлую ментальную модель.

`fagent` делает обратную ставку: меньше слоёв, уже фокус, выше операционная ясность.

## Где `fagent` выигрывает у OpenClaw и похожих систем

Это не абсолютные утверждения. OpenClaw шире. `fagent` собран плотнее.

### 1. Меньше операционная поверхность

`fagent` концентрируется вокруг:

- одного Python package
- одного workspace
- одного конфига под `~/.fagent`
- одного CLI для bootstrap, чата, gateway, auth, channels и memory tooling

Отсюда практические плюсы:

- быстрее онбординг
- проще увидеть config drift
- локальнее и понятнее отладка
- легче расширять рантайм маленькой команде

OpenClaw, напротив, публично делает ставку на более широкий platform surface: кроссплатформенные приложения, расширения, auth helpers и более обширную продуктовую форму. Для охвата это плюс, для простоты эксплуатации `fagent` удобнее.

### 2. Лучше подходит инженерам, которые хотят читать рантайм

`fagent` полезно opinionated:

- основной agent loop читается
- инструменты зарегистрированы как прямые runtime objects
- память реализована явными слоями, а не как непрозрачный black box
- `fagent onboard` создаёт полный конфиг, а не полу-шаблон

Это важно, когда нужно понять:

- почему агент вспомнил именно это
- откуда пришла конкретная память
- какая модель использовалась для shadow, embeddings или workflow repair
- почему шаг workflow перезапустился или эскалировался

### 3. Сильнее continuity в длинной работе

Многие агенты умеют отвечать. Намного меньше систем умеют стабильно продолжать реальную проектную работу без постоянного ручного напоминания контекста.

`fagent` добивается continuity через:

- file memory для читаемых долговременных заметок
- vector memory для семантического поиска
- graph memory для сущностей и связей
- workflow snapshots для текущего состояния решения
- task graph state для целей, блокеров и решений
- experience memory для повторяющихся operational patterns
- shadow-context compression перед вызовом основной модели
- auto-summary длинных сессий

Из-за этого `fagent` практически выигрывает у агентов, которые в основном живут на текущем conversation window плюс паре summary-слоёв.

### 4. Лучше дисциплина по стоимости и latency в tool-heavy сценариях

В `fagent` есть встроенный `run_workflow`, который умеет выполнять упорядоченные tool steps и подключать более лёгкую helper-модель для интерпретации и recovery.

Это значит, что основной модели не нужно:

- заново планировать каждый маленький шаг
- тратить дорогие токены на локальный tool repair
- снова и снова рассуждать об одном и том же operational error surface

Практический результат:

- ниже latency
- ниже стоимость
- меньше мусора в основном диалоге

## Как устроена память

Память в `fagent` многослойная, а не монолитная.

### Слой 1: file memory

Файловое хранилище пишет читаемые артефакты в workspace:

- `memory/HISTORY.md`
- `memory/MEMORY.md`
- `memory/daily/YYYY-MM-DD.md`

Это самый прозрачный слой памяти.

Плюсы:

- можно открыть руками и проверить
- хорошо подходит для аудита
- остаётся полезным fallback даже если embeddings или graph retrieval временно недоступны

### Слой 2: vector memory

Артефакты и эпизоды индексируются в vector backend для семантического recall.

Это помогает:

- при fuzzy retrieval
- когда вопрос переформулирован иначе, чем раньше
- когда нужно достать релевантную прошлую попытку, а не exact string match

Плюсы:

- устойчивее, чем keyword-only поиск
- лучше работает на грязных реальных формулировках
- не привязан к буквальному повторению текста

### Слой 3: graph memory

`fagent` записывает graph-oriented память для сущностей, отношений и evolving structure. Можно работать локально или через Neo4j.

Это критично для вопросов вроде:

- как связаны эти две вещи
- какое решение породило текущий blocker
- какая сущность относится к этому workflow

Плюсы:

- relationship recall не сводится к plain text search
- появляется continuity на уровне сущностей
- решения, блокеры и факты можно связно обходить

### Слой 4: workflow snapshots

Во время работы `fagent` может сохранять workflow snapshots, где есть:

- current state
- open blockers
- next step
- citations
- tools used

Это очень сильный слой, потому что continuity в реальной работе часто про состояние исполнения, а не просто про факты.

Плюсы:

- легче возобновлять работу после прерывания
- лучше восстановление после partially completed tasks
- понятнее продолжение отладки, миграций и repair-сценариев

### Слой 5: task graph state

Memory registry хранит task nodes и task edges для goals, steps, decisions и blockers.

Плюсы:

- агент помнит, что сейчас в процессе, что заблокировано и что уже решено
- continuity становится task-aware, а не только chat-aware
- состояние проекта переживает одно окно диалога

### Слой 6: experience memory

Когда operational patterns повторяются, `fagent` пишет experience patterns:

- что сломалось
- какое recovery сработало
- сколько раз этот паттерн уже встречался

Особенно полезно для повторяющихся runtime-ошибок и repair loops.

Плюсы:

- неудачи превращаются в повторно используемое знание
- агент перестаёт каждый раз заново открывать один и тот же фикс
- recovery может ранжироваться по evidence count, а не только по свежести

### Слой 7: shadow context

Перед запуском основной модели `fagent` может собрать память из file, vector и graph слоёв и сжать её через более лёгкую модель в компактный shadow brief.

В этом brief есть:

- summary
- facts
- open questions
- citations
- confidence
- contradictions
- store breakdown

Плюсы:

- основная модель получает сжатый рабочий контекст вместо сырого memory clutter
- проще контролировать token usage
- можно подсветить противоречия до того, как они испортят основной ответ
- слабое retrieval quality явно помечается как low-confidence

### Слой 8: query-aware routing

`fagent` не ходит в каждое хранилище одинаково для каждого запроса.

Memory router определяет intent запроса, например:

- temporal recall
- workflow recall
- relationship recall
- factual recall
- preference recall
- continuity
- broad synthesis

Дальше это влияет на:

- порядок и набор стораджей
- bias по artifact types
- возможность raw evidence escalation
- объём retrieval

Плюсы:

- меньше нерелевантных воспоминаний
- выше качество recall под конкретный тип вопроса
- меньше пустой нагрузки на контекст

## Почему такая память сильнее простых memory-подходов

Главный плюс — composability.

У простых агентов обычно один-два memory-механизма:

- только chat history
- chat history плюс embeddings
- только periodic summarization

`fagent` совмещает несколько форм памяти, каждая из которых закрывает свой failure mode:

- file memory решает прозрачность
- vector memory решает семантическую размытость
- graph memory решает связи
- workflow memory решает continuity активной работы
- experience memory решает повторяющиеся operational lessons
- shadow context решает compression
- routing решает relevance

Именно поэтому `fagent` может быть сильнее в continuity-heavy инженерных и операторских сценариях.

## Встроенный инструмент последовательных workflow

В `fagent` встроен отдельный `run_workflow` tool для последовательного выполнения шагов.

Он принимает:

- цель
- список ordered steps
- явный `allowed_tools`
- режим LLM assistance
- escalation policy

Он умеет:

- нормализовать rough step syntax
- выводить positional arguments
- выполнять шаги по порядку
- обнаруживать blocked и invalid steps
- повторять шаг с repaired payload
- обращаться к более лёгкой helper-модели за recovery guidance
- возвращать управление основной модели только когда это действительно нужно

## Почему этот workflow tool реально полезен

### 1. Он отделяет orchestration от большого reasoning

Основная модель решает, что нужно сделать. Workflow tool берёт на себя исполнение небольшой многошаговой tool-цепочки.

Это уменьшает waste и оставляет основной агент для более дорогого reasoning.

### 2. Он использует маленькую helper-модель там, где это разумно

`workflow_light` — отдельная model role. Благодаря этому можно назначить более дешёвую и быструю модель для:

- интерпретации неоднозначных шагов
- recovery suggestions
- repair после tool errors

Во многих operational сценариях это правильный tradeoff, потому что не каждая ошибка требует strongest model.

### 3. Он превращает tool failures в память

Когда workflow repair срабатывает, tool может записывать recovery telemetry в experience memory.

Это значит:

- ошибки становятся долговременным знанием
- агент узнаёт повторяющиеся паттерны
- в следующий раз recall может сразу поднять известный путь восстановления

### 4. Он делает escalation явной

Workflow tool может либо fail fast, либо вернуть управление основной модели.

Это лучше, чем тихие бесконечные retry loops, потому что можно точно настроить:

- когда локальный repair разрешён
- когда должна включаться основная модель
- какие tools вообще разрешены

## Практические плюсы для команды

`fagent` особенно хорош там, где важны:

- local-first debugability
- предсказуемая конфигурация
- recoverable long-running work
- контроль стоимости в tool-heavy flows
- прозрачная память, которую можно читать на диске
- понятный путь к добавлению tools, channels и model roles

## На чём основано сравнение

Часть про OpenClaw опирается на официальные публичные материалы:

- [OpenClaw GitHub repository](https://github.com/openclaw/openclaw)
- [OpenClaw documentation site](https://docs.openclaw.ai)

Вывод не в том, что `fagent` универсально лучше всего. Вывод в том, что `fagent` лучше оптимизирован для инженеров и операторов, которым нужен сфокусированный runtime с сильной continuity, слоистой памятью и меньшей операционной сложностью.
